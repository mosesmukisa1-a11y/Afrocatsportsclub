SELF-REGISTRATION DESIGN (IMPORTANT)

We need TWO flows:

FLOW 1: Player Self Signup (Recommended)
- Player registers with email+password AND chooses tenantSlug (e.g. "afrocat")
- Player then completes their Player Profile form
- System creates:
  - User role=PLAYER, tenantId resolved from tenantSlug
  - Player record (tenantId same)
  - links User.playerId -> Player.id
- Team assignment is NOT chosen by player by default (Admin/Manager assigns team later)
- Player is allowed to update only personal fields (phone, address, next-of-kin, photo, etc.)

FLOW 2: Admin adds player then invites/links login
- Admin creates Player record
- Admin creates User record and links user.playerId to the player

AUTH / API CHANGES

1) Registration endpoint must support player self signup:
POST /auth/register
Body:
{
  fullName,
  email,
  password,
  tenantSlug,
  role: "PLAYER" (default if not provided)
}
Server:
- find tenant by slug
- create user with tenantId + role=PLAYER
- create empty Player profile linked to user (firstName/lastName derived from fullName if possible)
- return token + user

2) “Complete profile” endpoint:
PUT /players/me
Body: editable fields + photoUrl
Rules:
- Player role can only update own player record

3) Admin/Manager player CRUD:
POST /players (admin)
PUT /players/:id (admin)
GET /players/:id (admin/coach with tenant/team rules)

4) Photo Upload
Implement:
POST /players/:id/photo (Admin/Manager OR Player if own)
- Accept multipart upload
- Save locally (dev) OR via Google Drive folder if configured (playerPhotoDriveFolderId)
- Save photoUrl

NOTE: If you already have SystemAssetConfig with playerPhotoDriveFolderId, use it.
