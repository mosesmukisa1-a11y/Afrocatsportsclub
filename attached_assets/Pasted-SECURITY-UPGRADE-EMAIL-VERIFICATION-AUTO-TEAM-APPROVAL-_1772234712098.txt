SERVER BEHAVIOR

A) On Registration (/auth/register)
1) Create user with:
- emailVerified = false
- accountStatus = PENDING_APPROVAL
2) Create player with:
- registrationStatus = PENDING_APPROVAL
3) Generate email verification token:
- random 32+ bytes token
- store hashed token + expiry (e.g., 24h)
4) Send verification email:
- Link: FRONTEND_URL + "/verify-email?token=...."
If SMTP not configured, fallback:
- Log the verification link in server console in dev mode
- Provide Admin UI to manually verify user (for early testing)

B) Verify Email Endpoint
POST /auth/verify-email
Body: { token }
Server:
- hash token and match user
- if valid and not expired:
  emailVerified=true, emailVerifiedAt=now
  clear token fields
- return success

C) Login rules
POST /auth/login must enforce:
1) If requireEmailVerification AND emailVerified=false => block login with message:
   "Please verify your email"
2) If requireAdminApproval AND accountStatus != ACTIVE => block login with message:
   "Your registration is pending approval"

D) Admin Moderation UI and endpoints
Create admin endpoints:
GET /admin/registrations/pending
- list pending users/players with requested team/position/jersey

POST /admin/registrations/:userId/approve
Actions:
- user.accountStatus = ACTIVE
- player.registrationStatus = APPROVED
- If auto-approve flags enabled, apply them (below)

POST /admin/registrations/:userId/reject
Body: { reason }
Actions:
- user.accountStatus = REJECTED
- player.registrationStatus = REJECTED
- store reason in player.registrationNotes

E) Auto Team Approval Rules
If SystemSecuritySettings.autoApproveTeamRequests=true:
- On admin approval OR on registration (if you choose), automatically set:
  player.teamId = requestedTeamId
  player.teamApprovalStatus = APPROVED
Else:
- keep teamApprovalStatus=PENDING until manually approved

Similarly:
- autoApprovePosition => set official position and approve
- autoApproveJersey => assign jersey only if team already approved and jersey not taken; else keep pending
